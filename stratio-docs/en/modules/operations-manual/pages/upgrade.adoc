= Version upgrade

== Description

The `upgrade-provisioner.py` script automates the upgrade of Kubernetes clusters in the following environments:

* *EKS* on Amazon Web Services (managed)
* *Azure VMs* on Microsoft Azure (unmanaged)
* *GKE* on Google Cloud Platform (managed)

It allows upgrading the Kubernetes cluster version from the one installed with `cloud-provisioner 0.7.X` to the version provided by `cloud-provisioner 0.8.X`.

To ensure reproducible execution, the Docker image `cloud-provisioner-upgrade:0.17.0-0.8.X` has been created. It includes the upgrade script and all required dependencies.

=== What gets upgraded

* *Common components* across all platforms:
** _cluster-operator_: `0.7.0-m.1`
** _cert-manager_: `v1.20.2`
** _flux2_: `2.18.3`
** _tigera-operator_: `v3.32.0`
** _cluster-autoscaler_: `9.52.1`

* *Cluster API Core*
** _cluster-api (CAPI)_: `v1.10.10`
** _bootstrap-kubeadm_: `v1.10.10`
** _control-plane-kubeadm_: `v1.10.10`

* *Platform-specific infrastructure providers*:
** CAPA (AWS): `v2.9.3`
** CAPZ (Azure): `v1.21.3`
** CAPG (GCP): `1.6.1-0.4.0`

* *Platform-specific charts*:
** EKS (if installed):
*** _aws-load-balancer-controller_: `1.14.1`
** Azure VMs:
*** _azuredisk-csi-driver_: `1.33.5`
*** _azurefile-csi-driver_: `1.34.1`
*** _cloud-provider-azure_: `1.34.2`

* Technical:
** Docker must be installed on the system.
** A _kubeconfig_ file with access to the cluster must be available (.kube/config).
** A _secrets.yml_ file (Ansible Vault) used during cluster creation must be available.

== Preparation

Create a local directory for backups:

[source,bash]
----
mkdir -p $PWD/backup
----

NOTE: This directory must be mounted inside the container at `/upgrade/backup`.

== Required structure

Ensure that the working directory includes:

* `upgrade-provisioner.py`: main script.
* `templates/`: Jinja2 templates.
* `requirements.txt`: required dependencies.
* `secrets.yml`: cluster credentials.
* `.kube/config`: _kubeconfig_ file for the cluster.
* `backup/`: directory for backups (may initially be empty).

== Important notes

=== Using the `--private` flag

The `--private` flag *forces* the use of private repositories for:

* Container images (`private_registry`).
* Helm charts (`private_helm_repo`).

The final behavior of the script does not depend solely on the flag, but also on the values already defined in `ClusterConfig`.

This means that:

* If `ClusterConfig.spec.private_registry=true`, the script will continue using the private registry even if `--private` is *not provided*.
* If `ClusterConfig.spec.private_helm_repo=true`, the script will continue using the private Helm repository even if `--private` is *not provided*.
* *If `--private` is provided*, both values are forced to `true` during the upgrade, and the `ClusterConfig` is updated accordingly.

==== Common scenarios

[%header,cols="2,2,3"]
|===
| Current state in `ClusterConfig`
| Execution
| Result

| `private_registry=true` +
`private_helm_repo=true`
| without `--private`
| The private registry and private Helm repositories will continue to be used.

| `private_registry=false` +
`private_helm_repo=false`
| without `--private`
| Public repositories will be used.

| `private_registry=false` +
`private_helm_repo=false`
| with `--private`
| The use of the private registry and private Helm repository is enforced, and `ClusterConfig` is updated to `true`.
|===

==== Chart and image sources

* When `private_helm_repo=true`, charts are resolved from `KeosCluster.spec.helm_repository.url`.
* When `private_registry=true`, Cluster API images are resolved from the registry marked as `keos_registry=true` in `KeosCluster.spec.docker_registries`.

WARNING: The `--private` flag is only required if you want to *force the use of private repositories* and the current `ClusterConfig` does not already have it enabled.

== Using the upgrade script

=== Syntax

Inside the container, run:

[source,bash]
----
python3 upgrade-provisioner.py -p <vault-password>
----

To see all available options:

[source,bash]
----
python3 upgrade-provisioner.py --help
----

Main options:

[%header,cols="1,2,1,1"]
|===
| Flag | Description | Default value | Required

| `-p`, `--vault-password`
| Specifies the Vault password required to decrypt secrets.
|
| Yes

| `-k`, `--kubeconfig`
| Specifies the _kubeconfig_ file to use.
| ~/.kube/config
| No

| `-s`, `--secrets`
| Specifies the encrypted secrets file.
| secrets.yml
| No

| `--disable-backup`
| Disables the pre-upgrade backup (enabled by default).
| False
| No

| `--disable-prepare-capsule`
| Disables environment preparation for the upgrade process.
| False
| No

| `--private`
| Indicates that the Docker registry and Helm repository are private.
| False
| No
|===

=== Running the upgrade container

[source,bash]
----
docker run \
  --name cloud-provisioner-upgrade-0.8.X \
  --net host \
  -it \
  -v $PWD/secrets.yml:/upgrade/secrets.yml \
  -v $PWD/.kube/config:/upgrade/.kube/config \
  -v $PWD/backup:/upgrade/backup \
  cloud-provisioner-upgrade:0.17.0-0.8.X
----

== Upgrade process overview

The script executes the following workflow:

. *Validation*
** Validates _kubeconfig_ and secrets.
** Configures cloud provider credentials.

. *Pre-upgrade*
** Scales _cluster-autoscaler_ to 0 to prevent automatic scaling during the upgrade. (Does not apply in GKE).

. *Backup*
** CAPX components (via `clusterctl move`).
** Capsule webhook configurations.

. *Capsule preparation*
** Modifies webhooks to exclude critical namespaces from validation and mutation.
** Ensures tenant isolation does not block the upgrade.

. *Chart upgrade*
** Upgrades base charts (cert-manager, flux2, tigera-operator, etc.).
** Upgrades provider-specific charts (aws-load-balancer-controller, Azure CSI controllers).
** Restores Capsule webhooks to their original configuration.

. *_Cluster-Operator_ preparation*
** Suspends the _HelmRelease_ of _cluster-operator_.
** Stops the _keoscluster-controller_ deployment.
** Disables _keoscluster_ validation/mutation webhooks.
** Updates _ClusterConfig_ with new component versions.

. *CAPI upgrade*
+
[source,bash]
----
clusterctl upgrade apply \
  --core cluster-api:v1.10.10 \
  --infrastructure <proveedor>:<versiÃ³n> \
  --wait-providers
----

. *Post-upgrade*
** Restores _keoscluster_ webhooks.
** Starts the _keoscluster-controller_ deployment.
** Resumes the _HelmRelease_ of _cluster-operator_.
** Waits until the _cluster-operator_ is _ready_.
** Waits for all components to reach the _ready_ state.
** Restores _cluster-autoscaler_ replicas to 2.
